# Celeborn

**Read [README.md](README.md) first — it has all the install and operating instructions.**

Celeborn must be **installed** (a CLI plus Claude Code hooks), not just copied into a project — copied
files do nothing on their own. Start at **[README.md → Install](README.md#install)**.

## Contact & identity — hard rule

Celeborn's public contact is the **support chat** (celeborncode.ai/faq and the board's Support
pill) — **chat only, no email**. Where an email address is genuinely essential (machine-readable
contact fields like llms.txt, package-registry metadata, chat-is-down fallbacks), the **only**
permitted address is **`hello@thot.ai`**. Never use `sales@thot.ai` in anything new, and never put a
personal or work email address — the maintainer's or anyone else's — anywhere associated with
Celeborn: marketing copy, source, docs, commit metadata, or the `.context/` tiers. Associating a
personal/work identity with this project creates real legal risk. If you find one, replace it with
`hello@thot.ai` (or redact it in historical/context records) and flag it.

## Task references — always use the full ID (CELE-tN)

In all human-readable text — conversation with the user, commit messages, journal entries, and notes
— always write a task by its **full project-qualified id** (e.g. `CELE-t122`), never bare `t122`. A
Celeborn fleet runs many projects' cards at once, so a bare `tN` is ambiguous about which project it
belongs to. The qualifier is the project's short prefix (`celeborn` → `CELE-`), exactly as the board
renders it. The only exceptions are machine-parsed contexts, which keep the bare `tN`: CLI arguments
(`celeborn claim t122`, `celeborn tasks edit t122 …`), commit trailers (`Celeborn-Task: t122`), and
card markers (`⟨celeborn:t122⟩`).

## Context — maintained by Celeborn

This project's long-term context lives in `.context/`, managed by Celeborn
(<https://github.com/cloud-dancer-labs/celeborn>). **Orient before acting:** read
`.context/state.md` (the headline) and `.context/session.json`; then `.context/notes.md`
(open threads, constraints, working detail) and `.context/durable/manifest.md`. Run
`celeborn search "<topic>"` to recall older details, and check `.context/journal.md` so you
don't redo finished work. As you make meaningful changes, record them back into the authored
tiers (state / journal / decisions / learnings) — that is what keeps the next session cheap.

**Multi-agent kanban:** When this project uses Celeborn tasks (`.context/tasks.md`), every
model sharing the repo sees the same board on orient — who's `doing` what. Celeborn is the
live source of truth for work in progress; external issue trackers are downstream reporting
only. Before taking a card, read the board (`celeborn tasks`) and choose a TODO that won't
interrupt another agent's in-flight work. Claim with `celeborn claim <id> --by <your name>` —
owner and DOING update for everyone. Pasting a copied card (its `⟨celeborn:tN⟩` marker) also
claims on receipt.

**Every card has a Stop condition:** each task carries a logical **Stop condition** (its `stop`
field) — a clearly-defined "this is a clean place to stop" marker that tells you when the card is at
a defensible `/clear` point. `celeborn tasks add` auto-fills a generic default so no card is ever
stop-less; when you claim a card, replace that default with a real one for the card: `celeborn tasks
edit <id> --stop "<condition>"`. Reaching the Stop condition is the signal that you may cleanly
`/clear`. `celeborn doctor` flags open cards with no Stop condition (or still on the default).

**Multi-agent file touches:** Before editing a shared file, register it so other agents see you
on orient: `celeborn touch <file> --task <id> --by <your name>`, then `celeborn touch release
<file>` when done. Never tag every line of code with agent metadata — touches + journal +
commit trailers (`Celeborn-Agent`, `Celeborn-Task`) are the protocol. Shorthand:
`references/multi-agent-editing.md`.

**Staying current:** Celeborn lives at <https://github.com/cloud-dancer-labs/celeborn>. Run
`celeborn version --check` now and then — it looks back at GitHub and tells you if a newer
Celeborn is available (and how to update).

**Surface context-health notices (but NOT the heartbeat):** the Celeborn `UserPromptSubmit`
hook injects two distinct channels. The **heartbeat** (`🏹 Celeborn —> M tokens recorded this
session · +N last turn`) is tagged *context only, do NOT surface* — it informs you, you do not
reprint it. As the context window fills, a **context-health notice** is also injected, tagged
*SURFACE THIS TO THE USER*: it first asks you to freshen the Hot tier (rewrite
`.context/state.md`, update `.context/session.json`, append `.context/journal.md`) so a `/clear`
or compaction loses nothing, then hands you a `🏹 Celeborn —>` line to relay. On surfaces like
the Claude desktop/web app the user never sees hook output — only your reply — so when that
notice is present: do the checkpoint, then begin your reply with the line(s) it gives you,
verbatim on their own lines, before anything else. Don't summarize or skip them.
